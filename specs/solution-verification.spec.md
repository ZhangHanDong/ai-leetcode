# Solution Verification Spec

## Goal

把候选思路变成可执行、可测试、可解释的实现。

## Required solution layers

1. 核心算法：与在线平台包装类型解耦；
2. 平台适配：只负责输入输出或签名转换；
3. 可复制提交：Rust 代码必须精确匹配平台模板，并且不依赖仓库内部模块；
4. 测试：直接调用核心算法，同时在模拟平台类型的测试夹具中编译可复制提交；
5. 可选 Trace：不得改变算法语义。

## Required checks

- 编译或语法检查；
- 校验可复制提交中的方法名、参数、返回值和 `impl Solution` 包装；
- 基础、边界、退化和对抗用例；
- 多实现使用同一组语义测试；
- 在适用时加入性质测试或与暴力基线交叉验证；
- 时间、空间复杂度包含推导和前提；
- 每一种算法都有 Mermaid 复杂度图，同时标出时间复杂度、额外空间复杂度、变量定义和成立前提；
- 图示计算当前实现的真实存储成本，不得遗漏输入物化、复制缓存或辅助索引；
- 正确性证明覆盖初始化、保持和终止。

## Acceptance criteria

- 所有固定测试通过；
- 书中展示的提交代码与测试编译的提交文件来自同一源文件；
- 复杂度图与正文推导、Rust 实现以及已有 benchmark 的适用条件一致；
- 优化解与基线在生成数据上输出一致；
- 不使用 benchmark 结果代替正确性验证；
- 问题状态可从 `draft` 推进到 `tested`，审阅证明后推进到 `verified`。
